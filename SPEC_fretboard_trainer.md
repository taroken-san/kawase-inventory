# フレットボード・トレーナー 引き継ぎ仕様書

## 0. 目的
ギターの指板把握（音名・度数の即答）を軸に、コードトーン・ガイドトーン・スケール・コード進行を「楽しく・耳でも」覚えられる学習アプリ。

## 1. 技術スタック
- **単一HTMLファイル**（依存ゼロ、CDN含めず）。ファイル名 `fretboard_trainer.html`
- **指板描画：Canvas**（`ukulele_tab_editor.html` と同系統の描画方針）
- **UI（モード切替・設定・スコア）：DOM**（Canvasの上にHTMLパネルを重ねるハイブリッド）
- **音：Web Audio API**（OscillatorNode、減衰エンベロープ付き。サンプル音源は使わない）
- 状態保存：`localStorage`（設定・ベストスコア・連続正解記録）
- iCloud Drive同期前提（保存先は別途たろけんさんが配置）

## 2. 確定パラメータ
| 項目 | 値 |
|---|---|
| チューニング | 標準（EADGBE）固定。内部的には可変設計にしておくが、UIは標準のみ |
| フレット数 | 既定15（設定で12/22に変更可能）。13〜15を含めることで「1フレットと13フレットが同じ音名」というオクターブ反復を視覚的に理解させるのが目的 |
| 度数表記 | ジャズ標準：R, b2, 2, b3, 3, 4, #11(=b5), 5, #5(=b6), 6, b7, 7。テンション表記 9/b9/#9/11/#11/13/b13 も内部で持つ |
| 異名同音 | Key基準で自動切替（例：G key なら F#、F key なら Bb）。シャープ/フラットはKeyの調号に従う |

## 3. データ構造

### 3.1 音高モデル
- 各弦の開放弦MIDIノート：`[40, 45, 50, 55, 59, 64]`（6弦E2→1弦E4）
- 任意の (string, fret) → MIDI = `openMidi[string] + fret`
- MIDI → pitchClass = `midi % 12`（0=C）

### 3.2 音名（Key依存スペリング）
- 12音を「シャープ系Key」「フラット系Key」で異なる表記辞書を用意
- Key選択（C〜B、12種＋主要なフラットKey）に応じてスペリングを決定

### 3.3 度数（Root依存）
- `interval = (pitchClass - rootPC + 12) % 12`
- interval→度数ラベル変換辞書（ジャズ標準）
- スケール文脈がある場合（後述）はスケール内のディグリーを優先表示

### 3.4 コード/スケール定義
```
chords: { maj7:[0,4,7,11], min7:[0,3,7,10], dom7:[0,4,7,10], m7b5:[0,3,6,10], dim7:[0,3,6,9], maj6:[0,4,7,9], min6:[0,3,7,9], ... }
scales: { ionian, dorian, mixolydian, aeolian, melodicMinor, ... }（半音間隔の配列で定義）
```

### 3.5 ボイシング辞書（コード学習の中核）
コードを「位置に依存しない形（シェイプ）」として持ち、ルート位置を指定すると指板上に展開する設計。

**データ表現：シェイプ＝弦ごとの相対オフセット**
- 各ボイシングを `{ strings: [...], degrees: [...], category, name }` で定義
- `strings`：低音側から、各弦の「ルートからのフレット相対値」または `null`（弾かない弦）
  - 例（Drop2 maj7 / 5弦ルート）：5弦=R, 3弦=7, 2弦=3, 1弦=5 のような配置を相対値で
- `degrees`：各発音弦が担う度数（R/3/5/7/9等）。色分け表示に使う
- ルート位置 (string, fret) を与えると、各弦の絶対フレット = ルートfret + 相対オフセットで算出
- 算出結果が 0〜maxFret に収まらない場合はそのポジションを「不可」として弾かない判定

**カテゴリ（タグ）**
- `shell`：シェル（R-3-7 / R-7-3）。4弦・3弦・2弦ルートの3バリエーション
- `drop2`：3つ閧／4声のドロップ2
- `drop3`：ドロップ3
- `freddie`：Freddie Green 系（3声、低音弦中心、ミュート前提）
- `iiVI`：ii-V-I で滑らかに繋がる実用ボイシング（声部の動きが少ないセット）

**辞書の網羅範囲（MVPで持つコードタイプ）**
- maj7, min7, dom7, m7b5, dim7（＋可能なら maj6, min6, 9th系）
- 各タイプについて上記カテゴリのボイシングを、主要なルート弦ごとに定義

**1,3,7（シェル）の扱い**
- たろけんさんの「1,3,7だけ弾く」要望に直結。shellカテゴリを最優先で実装
- ルート省略形（3,7のみ＝ガイドトーン）も shell の派生として持てる設計にしておく

**ボイシングの作り方（ハイブリッド方針・確定）**
- `shell` / `iiVI` / `freddie`：**手定義**。運指が良く実用的なものだけを厳選して辞書に持つ
- `drop2` / `drop3`：**自動生成＋フィルタ**。コード構成音とルールから展開し、以下の制約で実用ボイシングのみ残す
  - 最大フレットスパン 4以内
  - 隣接弦のみ使用（飛び越し弦は除外、または別扱い）
  - 各構成音が1回以上含まれる（ドロップ変形の定義に従う）
- 生成結果のうち、運指が破綻するものは除外。手定義リストとマージして表示

## 4. 機能スコープ

### MVP（軸＝音名・度数即答）— 最初に作り込む
1. **指板描画**
   - 6弦×15フレット、ナット・フレットマーカー（3,5,7,9,12,15のドット。12は二重ドット）
   - 各ポジションをタップ/クリック可能
   - レスポンシブ（横スクロール or 縮尺調整）。15フレットは横幅が出るので横スクロール前提で設計

2. **クイズモード A：位置→答える**
   - ランダムな1ポジションが光る → 音名 or 度数を選択肢から答える
   - 出題対象：音名モード / 度数モード を切替
   - 度数モードはRoot（=Key）を設定

3. **クイズモード B：答え→位置を探す**
   - 「Gを探せ」「b3を探せ」と提示 → 該当ポジションを全てタップ
   - 全正解で次へ

4. **自由探索モード**
   - タップすると音名・度数・音が出る。学習・確認用
   - Key/Root切替で度数表示が動的に変わる

5. **音**
   - タップ時・正解時に Web Audio で発音
   - 設定でON/OFF、音量調整

6. **スコア/ゲーム性**
   - 連続正解（ストリーク）、正答率、応答時間
   - localStorageにベスト記録保存
   - 軽い達成演出（色・アニメーション。過剰にしない）

### 第二フェーズ（コード学習 — 今回の追加要望の中核）
- **コードモード（ボイシング辞書）**
  - コードタイプ（maj7/min7/dom7/m7b5/dim7…）とルートを選択
  - そのコードで取り得るボイシングを一覧表示（カテゴリでフィルタ：shell / drop2 / drop3 / freddie / iiVI）
  - 1つ選ぶと指板上に展開。15フレット内で成立する複数ポジションを切替できる
  - **表示は両方切替**：
    - (a) 度数色分け表示（R/3/5/7/9を色分け）
    - (b) 鏃型（コードダイアグラム）表示
  - 各ボイシングを Web Audio でストロークまたはアルペジオ再生
  - **シェル（1,3,7）を最優先**。ルート省略の3-7ガイドトーン形も表示可能に
- **ガイドトーン**：2コード間の3rd・7thの最短移動をハイライト（コードモードの上に構築）
- **コード進行モード**：ii-V-I 等の進行に沿って、iiVIカテゴリのボイシングを声部の動きが最小になるよう繋いで提示

### 第三フェーズ
- **スケール/ポジション**：スケール選択→該当音を表示、ポジション（CAGED or 3note-per-string）で絞り込み

**生成の度数配置基準（ユーザー提供の基本形に準拠）**
- 基本コードフォーム（6弦/5弦ルートの R△7・R7・Rm7・Rm7b5・Rdim7・R6・シェル省略形）は**個別に辞書化せず、生成ロジックで再現する**。これらは度数配置が定型なので、コードタイプ×ルート弦の生成ルールとして実装する
- 生成時の「どの弦にどの度数を置くか」の基準配置は、ユーザー提供の基本形図に準拠する（例：6弦ルートR△7＝6弦R・5弦p5・3弦△3・2弦△7／5弦ルートR△7＝5弦R・3弦△3・2弦p5・1弦△7 など）
- メジャースケールのポジション（5フレット毎ブロック＋合体図、15フレットまで）も同様に**スケール定義＋ポジション分割ルールで生成**し、個別辞書化しない
- **手定義辞書に残すのは**、生成ルールでは出てこないユーザー独自の運指・特定曲用の凝ったボイシングのみ

**コード名表記ルール（ユーザー慣用に準拠）**
- メジャー系：R△ / R△7、マイナー：Rm（=R-）、セブンス：R7 等、ユーザーの基本形図の表記に合わせる

### 3.7 自作素材の取り込みワークフロー
たろけんさんが自作した指板図・ダイアグラム（画像/PDF、100超）を辞書に取り込むための運用。**著作権フリーな自作素材**なので制約なく活用できるが、量が多いため自動一括変換は精度面で行わず、人＋Claudeの対話で段階的に取り込む。

**前提**
- アプリ本体に画像読み込み機能は持たせない。取り込みはアプリ外（Claudeとの対話）で行い、結果を `VOICING_LIBRARY` への追記スニペットとして反映する
- これにより辞書フォーマットは「人が読み書きしやすい純データ」を保つ（§3.5〜3.6と整合）

**3段階フロー**
1. **仕分け**：100超を全部入れない。重複・自動生成でカバー可能な標準形（drop2/drop3の定番）・使用頻度の低いものを除外し、「自分ならではの良い運指/ボイシング」に優先度づけ
2. **変換**：優先度の高いものから数枚〜十数枚単位でClaudeに渡す。Claudeが押弦位置(string,fret)を読み取り→度数・コードタイプを判定→相対オフセット形式に変換。1枚ずつ確認しながら進め、手書き等の読みにくいものはClaudeが推測を提示しユーザーが正誤を返す
3. **追記**：変換結果を `VOICING_LIBRARY` 追記スニペットとして出力。§3.6の検算ロジックで二重チェック

**辞書フォーマット側の要件（取り込みやすさ確保）**
- 1ボイシング=1レコードで、押弦位置から機械的に生成できる形を維持（§3.5の `offsets`/`degrees`）
- `source: "manual"` で自作取り込み分を識別可能に
- バッチ追記してもID重複しないよう、id命名規則を保つ（例 `<chordType>_<category>_<rootString>str_<連番>`）

## 5. UI構成案
- 上部：モードタブ（探索 / クイズA / クイズB / コード / ガイドトーン / 進行 / スケール）
- 左 or 上：設定パネル（Key/Root、音名↔度数、フレット数、コード表示=度数色分け↔ダイアグラム、ボイシングカテゴリ、音ON/OFF、音量）
- 中央：Canvas指板
- 下部：スコア表示・出題プロンプト・回答UI

## 6. 配色・表示ルール
- ルート音：強調色（赤系）
- コードトーン（3,5,7）：色分け
- スケール音：淡色
- 度数ラベルはポジション上に小さく表示（音名モードでは音名）
- ダーク/ライト両対応は任意（まずダーク基調で）

## 7. 実装順序（推奨）
**第一フェーズ（MVP）**
1. Canvas指板描画（15フレット）＋座標↔(string,fret)変換
2. 音高/音名/度数モデルと辞書（Key依存スペリング、Root依存度数）
3. 自由探索モード＋Web Audio発音
4. クイズA → クイズB → スコア/localStorage

**第二フェーズ（コード）**
5. ボイシング辞書のデータ定義（shell最優先 → drop2 → freddie → iiVI → drop3）
6. コードモード：選択→展開→ポジション切替、度数色分け↔ダイアグラム両表示
7. ボイシングのWeb Audio再生（ストローク/アルペジオ）
8. ガイドトーン → コード進行モード（ii-V-I）

**第三フェーズ**
9. スケール/ポジション

## 8. 受け入れ基準
**MVP**
- 標準チューニング15フレットで全ポジションが正しい音名・度数を返す
- 1フレットと13フレットが同じ音名（同pitchClass）として正しく表示される
- KeyをまたいでF#/Gb等のスペリングが調号通りに切り替わる
- クイズA/Bが動作し、ストリーク・正答率が保存される
- タップで音が鳴る（ON/OFF可能）
- 単一HTMLで完結し、ファイルをダブルクリックで動作

**コードモード**
- maj7/min7/dom7/m7b5/dim7 でshell・drop2・freddie・iiVIのボイシングが指板に正しく展開される
- ルート位置を変えると全ボイシングが追従し、15フレット外は非表示になる
- 度数色分け表示とコードダイアグラム表示が切替できる
- シェル（1,3,7）と3-7ガイドトーン形が表示・発音できる

## 9. 既知の注意点
- 度数表記 #11 と b5 はジャズ文脈で使い分け。MVPでは表記辞書を一元化し、スケール/コード文脈実装時に文脈優先ロジックを足す
- Web Audioはユーザー操作後に `AudioContext` を起動（自動再生制限対策）
- 異名同音スペリングのテスト（全12Key×12音）を実装時に検証する
